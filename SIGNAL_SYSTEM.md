# Forge Master Signal System

## Overview

The Forge Master uses a file-based signaling system to communicate with worker Claude Code instances. This enables asynchronous coordination and status updates across multiple development streams.

## How It Works

### For Workers (Claude Code Instances)
1. **Create signals** using the helper script or manually
2. **Signal types**: `task_complete`, `need_help`, `update_ready`, `question`, `blocked`
3. **File format**: `signals/[instance_name]_[message_type]_[timestamp].md`

### For Forge Master
1. **Monitor signals** using the monitoring script
2. **Process messages** from workers
3. **Archive processed signals** to `signals/processed/`
4. **Coordinate responses** and task assignments

## Usage

### Creating Signals (Workers)

**Method 1: Using the helper script**
```bash
./create_signal.sh instance1 task_complete "Implemented user authentication"
./create_signal.sh instance2 need_help "Stuck on database connection"
./create_signal.sh instance3 update_ready "Feature branch ready for review"
```

**Method 2: Manual creation**
```bash
# Create a signal file manually
cat > signals/instance1_question_$(date +%Y%m%d_%H%M%S).md << EOF
# Worker Signal: question

**From:** instance1
**Type:** question
**Timestamp:** $(date)
**Branch:** feature/user-auth

## Message

Should I use JWT or session-based authentication?

## Status

- [x] Task in progress
- [ ] Task completed
- [ ] Need assistance
- [ ] Ready for review

## Files Modified

- src/auth/auth.js (in progress)

## Next Steps

Awaiting architectural decision from Forge Master

---
*Generated by instance1*
EOF
```

### Monitoring Signals (Forge Master)

**Single check:**
```bash
./monitor_signals.sh
```

**Continuous monitoring:**
```bash
./monitor_signals.sh continuous
```

## Signal Types

- `task_complete`: Worker has finished an assigned task
- `need_help`: Worker is blocked and needs assistance
- `update_ready`: Worker has updates ready for review
- `question`: Worker has a question or needs clarification
- `blocked`: Worker cannot proceed due to external dependency
- `merge_ready`: Worker's feature branch is ready for merge

## Signal Template

```markdown
# Worker Signal: [TYPE]

**From:** [INSTANCE_NAME]
**Type:** [MESSAGE_TYPE]
**Timestamp:** [TIMESTAMP]
**Branch:** [CURRENT_BRANCH]

## Message

[Detailed message content]

## Status

- [ ] Task in progress
- [ ] Task completed
- [ ] Need assistance
- [ ] Ready for review

## Files Modified

- [List of files changed]

## Next Steps

[What should happen next]

---
*Generated by [INSTANCE_NAME]*
```

## Integration with Development Workflow

1. **Assignment**: Forge Master assigns tasks to workers
2. **Progress Updates**: Workers signal progress and completion
3. **Issue Resolution**: Workers signal when blocked or need help
4. **Code Review**: Workers signal when ready for review
5. **Coordination**: Forge Master coordinates responses and next steps

This system enables efficient coordination across multiple parallel development streams while maintaining clear communication channels.

## Automation Opportunities

- **Automated Responses**: Forge Master can automatically respond to certain signal types
- **Task Queuing**: Queue up responses and actions based on signal priority
- **Metrics Collection**: Track worker productivity and common issues
- **Integration**: Connect with GitHub PR/issue creation for seamless workflow
